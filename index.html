Jóvenes a la E (JE4) — ¿Cómo se asignan las ofertas de programas?

En esta página se explica, en lenguaje sencillo, cómo funciona el modelo JE4 que decide cuántos cupos se financian en cada programa de educación superior ofertado por las instituciones de educación superior (IES).

1) ¿Qué hace el modelo JE4?

El modelo JE4 responde a una pregunta concreta: con el presupuesto disponible, ¿cuántos cupos se pueden financiar en cada programa que presentaron las IES?

Para hacerlo, el modelo:

  1. Toma las ofertas de programas que enviaron las IES (nombre del programa, número de semestres, cupos, costos, etc.).
  2. Calcula el costo de financiar una cohorte completa de cupos para cada programa, distribuyendo los pagos en los diferentes años en que se haría el gasto (vigencias).
  3. Organiza el presupuesto en bolsas (por tipo de programa, por IES, por nivel) y define los límites por año.
  4. Recorre las ofertas en un orden definido (para que el procedimiento sea claro y repetible) y asigna cupos hasta donde lo permite el presupuesto y los topes de cada vigencia.

Resultado global

Al finalizar, JE4 produce una tabla donde, para cada programa habilitado, se indica: cuántos cupos se aprobaron y cómo se distribuye el gasto por año.

2) ¿Qué entendemos por “oferta” en JE4?

En JE4, una oferta es la combinación de:

  • Una institución de educación superior (IES).
  • Un programa específico (por ejemplo, Ingeniería X, Tecnología Y).
  • Un nivel del programa (Universitario, Tecnológico, Técnico profesional).
  • Un número de cupos que la IES propone financiar por medio del programa.
  • Los costos por semestre para las personas beneficiarias.

Antes de JE4, las ofertas ya han pasado por un proceso de verificación y están marcadas como “Habilitadas”; es decir, cumplen las condiciones mínimas exigidas por la convocatoria.

3) Presupuesto y bolsas de recursos

El punto de partida del modelo es un presupuesto general de la convocatoria. A partir de este valor se construyen varias “bolsas” o subconjuntos de recursos:

  • Una bolsa para el mecanismo actual (el que financia las ofertas analizadas en JE4).
  • Una bolsa para otros usos, como demanda social.
  • Dentro del mecanismo actual, una parte se reserva para programas universitarios.
  • Otra parte se reserva para programas técnicos y tecnológicos (TyT).

Ejemplo ilustrativo en R (simplificado):

PPTO_GENERAL <- 134411135364               # Presupuesto total (ejemplo)
PPTO_MecanismoActual <- PPTO_GENERAL * .9  # 90% al mecanismo actual
PPTO_DemandaSocial   <- PPTO_GENERAL * .1  # 10% a demanda social

# Dentro del mecanismo actual:
PPTO_UNIV <- PPTO_MecanismoActual * .95    # 95% para universitarios
PPTO_TYT  <- PPTO_MecanismoActual * .05    # 5% para técnicos y tecnológicos

Estas proporciones pueden ajustarse según las reglas de la convocatoria y las decisiones de política pública. Lo importante es que el modelo JE4 respeta estos límites al asignar cupos.

4) Costos por semestre y estructura de pagos

Cada programa tiene un costo por semestre. JE4 transforma esa información en una estructura de pagos que permite saber cuánto se paga en cada semestre y qué proporción del pago corresponde a cada vigencia.

Ideas clave:

  • Se consideran los semestres del programa (1, 2, 3, …).
  • Para cada semestre se calcula un pago 1 y, en algunos casos, un pago 2.
  • Cierta fracción (por ejemplo 80%) puede pagarse en el año en que se cursa el semestre, y otra fracción (por ejemplo 20%) se paga en una vigencia posterior.

Fragmento ilustrativo (no ejecutable por sí solo):

# La tabla OFERTA_COSTO tiene columnas de costos por semestre
OFERTA_COSTO <- pivot_longer(
  OFERTA_COSTO,
  cols      = 5:18,                # columnas de semestres
  names_to  = "SEMESTRE",
  values_to = "COSTO_SEMESTRE"
)

# Se define para cada semestre la proporción del pago 1 y pago 2
OFERTA_COSTO_EJECUCION$Pago1 <- 1
OFERTA_COSTO_EJECUCION$Pago2 <- .2

# En semestres pares el pago 1 es 80% y el pago 2 el 20% restante
OFERTA_COSTO_EJECUCION[SECUENCIA_SEMESTRES %in% c(2,4,6,8,10,12),"Pago1"] <- .8
OFERTA_COSTO_EJECUCION[!(SECUENCIA_SEMESTRES %in% c(2,4,6,8,10,12)),"Pago2"] <- 0

# Se pasa la tabla a formato largo (Pago1 / Pago2)
OFERTA_COSTO_EJECUCION <- pivot_longer(
  OFERTA_COSTO_EJECUCION,
  cols      = 9:10,
  names_to  = "NUMERO_PAGO",
  values_to = "PORCENTAJE_PAGO"
)

El resultado de este paso es una tabla que indica, para cada programa, semestre y tipo de pago, qué porcentaje del costo se paga y en qué momento.

5) Costos por año de ejecución (vigencias)

A partir de la estructura de pagos por semestre, JE4 calcula cuánto se gasta en cada año (vigencia). Esto es clave porque el presupuesto también tiene límites por vigencia.

En términos sencillos:

  1. Se identifica en qué año se ejecuta cada pago de cada semestre.
  2. Se multiplica el costo del semestre por la proporción del pago (Pago1, Pago2, etc.).
  3. Se suman esos valores por año para obtener el costo anual de la cohorte.

Idea principal en pseudocódigo:

OFERTA_COSTO_EJECUCION_ANUAL <- OFERTA_COSTO_EJECUCION |>
  group_by(COD_IES, COD_PROGRAMA, ANIO_SEMESTRE_EJECUCION) |>
  summarise(
    COSTO_ANUAL = sum(COSTO_SEMESTRE * PORCENTAJE_PAGO),
    .groups = "drop"
  )

Esta tabla permite saber, para cada programa, cuánto costaría una cohorte completa por año, y así verificar que la asignación de cupos no supere el presupuesto disponible en ninguna vigencia.

6) Bolsas por IES y nivel (TIR ponderada)

El siguiente paso es decidir qué parte del presupuesto por nivel le corresponde a cada IES. Para esto se usa una medida de retorno/pertinencia de los programas, de manera agregada.

Para cada combinación IES × nivel se calcula:

  • La suma de un indicador de retorno (por ejemplo, la TIR del programa).
  • La participación de cada programa en esa suma.

Fragmento ilustrativo de la TIR ponderada:

# Agrupación por IES y nivel
TIR_PONDERADA <- OFERTA_GENERAL |>
  group_by(COD_IES, NIVEL_PROGRAMA_SNIES_AJUSTE) |>
  summarise(TOTAL_TIR_IES_NIVEL = sum(TIR_PROGRAMA),
            .groups = "drop")

OFERTA_GENERAL <- merge(
  OFERTA_GENERAL,
  TIR_PONDERADA,
  by = c("COD_IES","NIVEL_PROGRAMA_SNIES_AJUSTE")
)

OFERTA_GENERAL$TIR_PARTICIPACION <- OFERTA_GENERAL$TIR_PROGRAMA /
  OFERTA_GENERAL$TOTAL_TIR_IES_NIVEL

Con esta información, la bolsa de presupuesto para cada IES×nivel se reparte entre sus programas de manera proporcional a la participación de su TIR.

7) Ordenamiento de programas dentro de cada IES

Dentro de cada IES×nivel, los programas se recorren siguiendo un orden que se define con criterios de calidad, costo y capacidad. Esto es importante para que la asignación sea consistente y reproducible.

En términos generales, el orden se basa en:

  1. Un indicador de calidad/pertinencia del programa (por ejemplo, ISOES, de mayor a menor).
  2. El costo de la cohorte (por ejemplo, menor costo primero, si se busca eficiencia).
  3. El número de cupos ofertados.
  4. Un desempate aleatorio controlado mediante una semilla fija (set.seed()).

Ejemplo de ordenamiento (simplificado):

set.seed(20251010)  # Semilla para desempates reproducibles

ORDENAMIENTO_PROG <- OFERTA_GENERAL |>
  arrange(
    COD_IES,
    NIVEL_PROGRAMA_SNIES_AJUSTE,
    desc(ISOES_PROGRAMA),
    TOTAL_VALOR_COHORTE_ATENEA,    # costo de cohorte
    desc(CUPOS_SEGUN_CAPACIDAD),   # más cupos primero
    SEMILLA                        # desempate aleatorio
  )

8) Asignación de cupos: recorrido del algoritmo

Una vez definido el orden, el modelo JE4 recorre los programas de cada IES×nivel y decide cuántos cupos se pueden aprobar, respetando tanto la bolsa global de la IES×nivel como los topes anuales de gasto.

Esquema conceptual:

  1. Para la IES×nivel se define un valor disponible de presupuesto (VALOR_IES).
  2. Para cada programa (en el orden definido), se calcula el cupo máximo teórico que permitiría ese presupuesto: CUPO_CALCULADO = VALOR_IES / TOTAL_VALOR_COHORTE_ATENEA.
  3. Se compara ese cupo con los cupos ofertados por la IES y con los topes por vigencia.
  4. Se aprueba el número de cupos que cumple las reglas sin exceder el presupuesto en ningún año.
  5. Se descuenta el costo de esos cupos del presupuesto disponible y se avanza al siguiente programa.

Pseudocódigo simplificado del recorrido:

for cada (COD_IES, NIVEL) {
  valor_disponible <- PPTO_IES_NIVEL

  for cada PROGRAMA en orden_ies_nivel {
    costo_cohorte <- TOTAL_VALOR_COHORTE_ATENEA

    # Cupo máximo por presupuesto
    CUPO_CALCULADO <- floor(valor_disponible / costo_cohorte)

    # Cupo máximo permitido por vigencias (mínimo por año)
    CUPO_VIG_MIN <- min(CUPOS_ANIO)   # derivado de OFERTA_COSTO_EJECUCION_ANUAL

    # Cupo ofertado por la IES
    CUPO_OFERTADO <- CUPOS_SEGUN_CAPACIDAD

    # Se aplica la regla de decisión (ejemplo):
    cupo_aprobado <- min(CUPO_CALCULADO, CUPO_OFERTADO, CUPO_VIG_MIN)

    if (cupo_aprobado > 0) {
      costo_total_programa <- cupo_aprobado * costo_cohorte
      valor_disponible <- valor_disponible - costo_total_programa
      registrar_aprobacion(PROGRAMA, cupo_aprobado, costo_total_programa)
    }
  }
}

Las reglas reales son más detalladas y contemplan casos particulares (por ejemplo, programas con costos en vigencias muy futuras), pero la idea central es siempre: no superar el presupuesto disponible ni los topes por año.

9) Resultados y salidas del modelo JE4

Al finalizar la ejecución, el modelo JE4 produce, para cada programa habilitado:

  • El número de cupos aprobados.
  • El costo total asociado a esos cupos.
  • La distribución del gasto por vigencia (año a año).

Estos resultados se consolidan en tablas y reportes que sirven tanto para la gestión interna como para la publicación de información a la ciudadanía, garantizando trazabilidad y transparencia.

10) Reproducibilidad y control del proceso

El modelo JE4 está implementado en R, utilizando principalmente los paquetes dplyr, tidyr, readxl y sqldf, entre otros.

Para garantizar reproducibilidad se aplican varias prácticas:

  • Uso de semillas fijas (set.seed()) para cualquier componente aleatorio (por ejemplo, desempates).
  • Control de versiones del código y de los insumos (archivos de oferta, costos y parámetros).
  • Conservación de los parámetros clave (presupuesto, proporciones, fechas) en archivos de configuración.

11) Preguntas frecuentes

¿Por qué un programa puede recibir menos cupos de los que ofertó la IES?

Porque el modelo JE4 debe respetar el presupuesto disponible y los topes por vigencia. Si financiar todos los cupos ofertados supera alguno de esos límites, solo se aprueba el número de cupos que sí se puede financiar sin exceder el presupuesto.

¿Cómo se decide el orden en que se revisan los programas?

El orden se define con reglas fijas basadas en calidad, pertinencia y costo (por ejemplo, un indicador como ISOES y la TIR ponderada). Cuando hay empates se utilizan semillas aleatorias controladas, para que el mismo código siempre produzca el mismo resultado.

¿Es posible replicar los resultados fuera de la Agencia?

En principio sí: con acceso al mismo código, los mismos insumos y los mismos parámetros (por ejemplo, presupuesto y semillas), se pueden reproducir los resultados paso a paso. Por motivos de protección de datos y licencias, algunos insumos solo pueden circular en entornos controlados.

12) Glosario

  • IES: Institución de Educación Superior.
  • Cohorte: grupo de cupos de un programa que se financian en una misma “oleada”.
  • Vigencia: año fiscal en el que se ejecuta un gasto público.
  • Cupo: oportunidad de matrícula financiada para una persona en un programa específico.
  • ISOES: indicador que resume información sobre calidad y oportunidades de educación superior.
  • TIR de programa: medida de retorno asociada al programa (usada para ponderar la asignación de recursos).

13) Notas técnicas (para personas que usan R)

El modelo JE4 se implementa en un script en R que:

  • Lee y depura los archivos de oferta (LISTADO_OFERTA_PROPUESTA y relacionados).
  • Construye tablas de costos por semestre, pagos y vigencias.
  • Define bolsas de presupuesto por nivel e IES.
  • Ordena y recorre programas, actualizando cupos y presupuestos.

Paquetes principales:

library(dplyr)
library(tidyr)
library(readxl)
library(readr)
library(sqldf)

Los detalles de implementación, consultas SQL y estructuras intermedias se documentan en los anexos técnicos y en el propio código fuente de JE4, dirigidos a equipos técnicos que necesiten auditar o extender el modelo.
